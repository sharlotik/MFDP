* Проект №1:
* Прогнозирование перечня и количества единиц для товаров закупки для магазинов одежды.

* 1. Ключевая гипотеза:

Если мы предлагаем продукт по прогнозу количества товаров для закупки,
то мы получим сокращение объема нереализованных товаров >= 10% и 
рост продаж востребованных продуктов  >= 10% в течение 3 месяцев, потому что 
наш сервис направлен на увеличение дохода от продаж за счет оптимизации закупок
и снижения упущенной выручки из-за отсутствия нужных товаров.

* 2. Ключевые метрики:

1.  Технические метрики: WAPE (Weighted Average Percentage Error) > 50%, 
Процент сокращения количества единиц нереализованных товаров и роста продаж востребованных товаров.
Для продаж вопстрабованных товаров будем использовать оценку: среднее количество продаж товара в день* количество дней с отсутствием товара в виду вродаж всех единиц до проведени следующей закупки.
2. Продуктовые метрики: Оценка дополнительной выручки от оптимизации прогноза (что в дальнейшем поможет оценить стоимость подписки на сервис), 
сокращения времени компании на формирование списка закупок 
3. Baseline: Текущие закупки, формируемые сотрудниками магазина: оценка количества покупок, которые не произошли из-за отсутствия товаров, и товаров, которые не можем продать длительное время.
4. Ожидаемое качество для технических метрик: средняя ошибка WAPE по всем SKU < 50%.
5. Ожидаемое качество для продуктовых метрик: Сокращение объема нереализованных товаров >= 10% и 
рост продаж востребованных продуктов >= 10%. Рост выручки по востребованным товарам и сокращение недополученной прибыли по плохо реализуемым товарам >= 10%.
6. Частота измерения: на основании оценки необходимого количества товаров, чтобы сделать вывод о приросте продуктовых метрик и каждая неделя для оценки технических метрик.

* 3. Что войдет/не войдет в прототип:
* Войдет в прототип:
Модель ассортимента закупаемых товаров + количества штук и размерно ряда. Краткий комментарий повыбору. 
Дашборд с оценкой эффекта для бизнеса. 
Сценарий выбором конкретного перечня товаров и количеством штук для проведения закупки, исходя из планируемого бюджета. 
Мы хотим протестировать базовый фанционал и проверить наличие реального эффекта для бизнеса, качество моделей.

* Не войдет в прототип: 
1. Прогнозирование/комментарии с помощью сложных моделей нейронных сетей
2. Анализ продаж других ритейлеров для советов по расширению ассортимента
3. Настройка интеграции с API магазина
Данные активности избыточны с точки зрения проверки гипотезы.

* 4. Что возьмем как готовые блоки для прототипа, что будем делать сами:
 Готовые блоки для прототипа: модель Catboost.
Сделаем сами: feature engineering, обучим CatBoost. 
Ранжирование товаров, исходя из суммы закупок.

* 5. Главный риск, который проверяем прототипом, и конечный вид прототипа:
* Главный риск:
1. Низкое качество моделей: WAPE > 50%
2. Недостачно данных для обучения
3. Низкое value для бизнеса: Не сможем дать прирост в >=10% к продажам востребованных продуктов и сокращение закупок долго реализуемых товаров на >= 10%

* Конечный вид прототипа: 
Таблица с перечнем товаров и количеством для проведения закупки.
Оценка дополнительных продаж и сокращения залежавшихся товаров в штуках и выручке по результатам цикла до следующей закупки. Дашборд.
Прототип будем проводить по историческим данным закупок и продаж в случае, если закупка проводится один раз за сезон, чтобы быстрее получить результат.

* 6. Критерии остановки прототипа:
1. WAPE > 60%, разница между baseline и модельным прогнозом < 5%
2. Эффект для бизнеса < 5% по дополнительным продажам востребованных товаров, которые ранее закупались в меньшем объеме, а также <5% сокращение количества плохо реализуемого товара. 


* Проект №2:
* Определение оценок и эмоций по отзывам о посещении организаций (данные с Яндекс.Карт + с сайта компании). 
* Классификация причин негитивных/позитивных отзывов (цена, персонал, скорость, доставка, качество, чистота).
* Оценка вероятности написания реального отзыва.

* 1. Ключевая гипотеза:

Если мы предлагаем продукт по прогнозу оценок и эмоций по отзывам для компаний,
то мы сокращаем время, которое Product Manager тратит на эти задачи, на >= 20% в течение первого месяца 
использования продукта, за счет автоматизации анализа отзывов. 

* 2. Ключевые метрики:

1. Техниеские метрики: 
Качество модели: МАЕ для высоких (4-5) и низких (1-2) оценок, F1-score для эмоций и классификации причин.
Способ измерения для оценок: сравнение прогноза с оценками пользователей.
Способ измерения для эмоций/причин: разметка с помощью LLM + проверка в ручном режиме 200-500 отзывов, по которым будем измерять качество модели.
2. Продуктовые метрики: Изменение времени, которое Product Manager тратит на анализ отзывов и выявление инсайтов. Способ измерения: засекать время работы над задачей
3. Baseline для технических метрик: Самая часто встречающаяся оценка, эмоция, причина для негативных и позитивных отзывов
4. Ожидаемое качество для технических метрик: 
    МАЕ <= 0,8 по оценкам, 70% F1-score по эмоциям и причинам
5. Ожидаемое качество для продуктовой метрики: сокращение времени работы Product Manager'а 
над анализом отзывов на >= 20%    
6. Частота измерения: еженедельно для технических метрик, раз в 2 недели для продуетовой метрики.

* 3. Что войдет/не войдет в прототип:
* 1. Войдет в прототип: 
Базовая модель оценок + эмоций + классификация отзывов + базовая фильтрация реальных отзывов + базовый дашборд.
Хотим проверить, что основной функционал приносит экономию времени работы для сотрудника.
* 2. Не войдет в прототип: 
1. Обучение моделей эмоций с помощью сложных моделей
2. Обработка сарказмов
3. Интеграция через API с CRM клиента

* 4. Что возьмем как готовые блоки для прототипа, что будем делать сами: 
* Возьмем для прототипа модель CatBoost, простые методы для векторизации (TF-IDF, Word2Vec)
* Сделаем сами: предобработку данных, создание эмбеддиногов/векторизацию, обучение CatBoost. 
Критичность в том, чтобы мы проанализировали данные перед обучением, обработали аномалии, 
обучились именно на используемом датасете.

* 5. Главный риск, который проверяем прототипом, и конечный вид прототипа:
* Главный риск:
1. Создание модели не приведет к сокращению работы Product Manager над задачей анализа отзывов более чем на 20%.
2. Получим низкое качество моделей

* Конечный вид прототипа: 
После обучения моделей возьмем новые отзывы, которые оценим и разметим по эмоциям, причинам, сравним со временем, сколько бы это дела Product в ручном режиме.

* 6. Критерии остановки прототипа:
1. Низкое качество технических метрик: 
MAE оценок > 1.2, разница между константным прогнозом оценки и модельным < 5%.
F1-score < 0.5 для причин и эмоций. 
2. Экономия времени Product Manager < 10% либо < 1ч.
