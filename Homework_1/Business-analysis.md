**1\.  Бизнес-анализ (Business Understanding)**  
** Цели и скоуп проекта.

Проект по прогнозированию ассортимента закупаемых товаров и количества каждого изделия.
Рассматривается как платформа, на которой магазины смогут получить прогноз по закупкам 
(предполагается начать с прогнозов для магазинов одежды).


* *Организационная структура: кто участвует в проекте со стороны заказчика сервиса, кто будет основным пользователем?* 

Заказчиком сервиса могут выступать ритейлеры, у которых есть потребность в планировании закупаемых товаров. 

В настоящий момент проект носит учебный характер, заазчиком выступает исполнитель, но при успешной реализации будут рассмотрены возможности:
1. Внедрения сервиса в действующую платформу для ритейлеров одежды, на которой они получают аналитику по продажам. В таком случае заказчиком будет выступать владелец платформы, пользователями - магазины, ведущие закупки.
2. Создания отдельной платформы для ритейлеров. Заказчиком будет выступать исполнитель, пользователями - магазины.

* *Собираем контакты, создаем рабочие чаты.*  

В случае внедрения в платформу будут созданы чаты с сотрудниками платформы и исполнителями.

* *Какова бизнес-цель проекта? Например, уменьшение оттока клиентов.*  

Бизнес-цель проекта состоит в предоставлении сервиса, позволяющего компаниям максимизировать 
прибыль от продаж востребованных продуктов, заказывая их в большем объеме (снижение дефицита), 
и сократить количество изделий, которые не востребованы покупателями (минимизация излишков), рост рентабельности инвестиций в запасы.

* *Существуют ли какие-то уже разработанные решения? Если существуют, то какие и чем именно текущее решение не устраивает?*

Текущие решения:
1. Forecast NOW!: прогнозирование спроса и оптимизации товарных запасов. 
2. Odyssey4Retail: консалтинговая платформа, предоставляющая в том числе услуги управления ассортиментом.
3. Imredi: платформа для работы в несольких сферах, включающих одежду. Решает задачу анализа пперемещения товаров по сети магазинов.
4. BI.Qube: оптимизация планов продаж для разных отраслей, работает с данными из 1С.

Текущее решение изначально будет будет узкоспециализированным на сфере работы с магазинами одежды,
работать в том числе с небольшими компаниями, в которых нет аналитиков и Data Scientists. 

**1.1 Текущая ситуация (Assessing current solution)**  
Оцениваем, хватает ли ресурсов для проекта.

* Есть ли доступное железо или его необходимо закупать?  

Доступное железо имеется, закупка не требуется.

* Где и как хранятся данные, будет ли предоставлен доступ в эти системы, нужно ли дополнительно докупать/собирать  
* внешние данные?  

Данные будут храниться в облачном хранилище, дополнительная закупка внешних данных будет рассматриваться после успешного запуска проекта.

* Сможет ли заказчик выделить своих экспертов для консультаций на данный проект?

При условии внедрения сервиса на платформу, заказчик будет проводить консультации.

Нужно описать вероятные риски проекта, а также определить план действий по их уменьшению.

Типичные риски следующие.

* Не уложиться в сроки.  
* Малое количество или плохое качество данных, которые не позволят получить эффективную модель.  
* Данные качественные, но закономерности в принципе отсутствуют и, как следствие, полученные результаты не  
* интересны заказчику.

Риски:
1. Низкое качество моделей
2. Недостачно данных для обучения
3. Низкая конвертация в подписку на услугу (при монетизации).

**1.2 Решаемые задачи с точки зрения аналитики (Data Mining goals)**  
Выполняем постановку в технических терминах. Для этого нужно ответить на следующие вопросы:

* Какую метрику мы будем использовать для оценки результата моделирования (а выбрать есть из чего: Accuracy, RMSE,  
* AUC, Precision, Recall, F-мера, R2, Lift, Logloss и т.д.)?  
* Каков критерий успешности модели (например, считаем AUC равный 0.65 — минимальным порогом, 0.75 —  
* оптимальным)?  
* Если объективный критерий качества использовать не будем, то как будут оцениваться результаты?

Будем усреднять MAPE каждому товару для двух случаев:
1. Перепрогнозирование (привело к большим закупкам, товары остались на складе) 
2. Недопрогнозирование (недополученная прибыль)

Ожидаемое значение МАРЕ от 0.7 до 0.8.

Без данного критерия можно будет померять выручку компании по сравнению с АППГ, предыдущим месяцем, трендом. Ожидаем рост дохода от продаж. 

**1.3 План проекта (Project Plan)**  
Как только получены ответы на все основные вопросы и ясна цель проекта, время составить план проекта. План должен  
содержать оценку всех шести фаз внедрения.  

1. Бизнес-анализ: формулируем цель проекта, метрики качества, риски.
2. Анализ данных: получаем исторические данные по продажам, оцениваем их достаточность, 
качество. Подготавливаем данные для моделирования (убираем аномалии, обрабатываем пропуски).
Генерируем новые признаки.
3. Моделирование: выбираем алгоритм для обучения (в качестве начального алгорима планируется взять CatBoost). Обучаем модель. Оцениваем качество.
4. Оценка решения: анализ достижения бизнес-целей (оцениваем эффект для бизнеса), этапов разработки (что можно было сделать по-другому). Оценки возможности улучшения модели и что для этого требуется сделать.
5. Внедрение: ввод модели в эксплуатацию. Создание дэшборда, интерфейса.
6. Тестирование и мониторинг работы модели.
